architecture

How it works.

Polling-based, stateful, multi-pipeline. Memory and skills layer added over Simba v2.

Telegram (polling)
    │
    ▼
telegram-bot.js
  - webhook drain + 409 conflict auto-retry (up to 60s)
  - message + callback_query routing
    │
    ▼
command-router.js
  - admin whitelist check
  - routes all commands
    │
    ├── /remember · /recall · /forget
    │      ▼
    │   memory.js :: MemoryManager
    │     persist facts to bot-state.json[memory{}]
    │
    ├── /skill define · run · list · delete
    │      ▼
    │   skills.js :: SkillsManager
    │     built-in: readme-gen · sql-migration · tool-spec
    │     custom skills in bot-state.json[skills{}]
    │     invoke → builds description → runUserPipeline()
    │
    ├── /task (plain description)
    │      ▼
    │   execution-pipeline.js :: runUserPipeline()
    │     ↳ injects memory context before GENERATE stage
    │     PLAN → GENERATE (Antigravity/Gemini) → ARTIFACT → send file
    │
    ├── /improve, /task (with repo:)
    │      ▼
    │   execution-pipeline.js :: runGitHubPipeline()
    │     PLAN → AUDIT → GENERATE → ARTIFACTS → PUSH → PR
    │
    └── /loop start
           ▼
        task-loop.js :: startLoop()
          continuous: generateNextTask → runGitHubPipeline → repeat

state-engine.js    — file-based state + memory{} + skills{} fields
memory.js          — MemoryManager · per-user key/value · context injection
skills.js          — SkillsManager · 3 built-ins · custom templates · {{input}}
github.js          — GitHub REST API (repos, branches, commits, PRs)
templates.js       — Codex prompt + repair prompt builders
artifacts.js       — Disk artifact writer + changes.patch generator
task-parser.js     — YAML/JSON/inline task parser + sanitization

Get running

Clone, install, configure two env vars, and start. Memory and skills are automatic — no config needed.

bash · quick start
# Clone and install
git clone <repo>
cd decide-engine-bot
npm install
cp .env.example .env

# Fill in .env then start
npm start

# Run tests (no Telegram needed)
npm run test:flow

key features

What makes
GN8R different.

Every feature ships from Simba v2, the new dual-pipeline layer, or the memory + skills update.

New
🧠
Persistent Memory
Store facts once with /remember — GN8R injects them into every future /task automatically. No more repeating your stack, preferences, or context.
New
⚡
Skills Layer
3 built-in skills (readme-gen, sql-migration, tool-spec) plus unlimited custom templates. Define once with /skill define, run anywhere with {{input}}.
📎
Real File Upload
Uses Telegram's sendDocument API to deliver actual files — not code blocks in chat. HTML, Python, Markdown, JSON, CSV, SQL, TypeScript.
🔀
Dual Pipeline Routing
Plain /task descriptions route to the user pipeline. Tasks with repo: field auto-route to the GitHub pipeline. One command, two outcomes.
🔒
File-Based State + Write Lock
State persists to disk with write locking and corrupt recovery. Memory and skills stored in the same JSON — survives bot restarts.
🔁
Continuous Task Loop
/loop start runs a continuous execution loop — generates the next task, runs the pipeline, repeats. Dry-run and live modes supported.
